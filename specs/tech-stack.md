# Swan-Admin Tech Stack

## 目标

本文件定义 Swan-Admin 前端管理后台的技术选型、职责边界与实现约束，用于避免重复选型、无序扩展和模板残留继续扩散。

本文件只约束前端后台，不覆盖后端 API、行情服务、数据库或基础设施方案。

## 技术栈基线

当前项目以前端后台为中心，采用以下技术基线：

- `React 19`：页面与组件开发的核心运行时。
- `TypeScript`：类型系统与工程约束基础。
- `Vite`：本地开发、构建与打包工具。
- `shadcn/ui`：后台通用组件基础层。
- `Tailwind CSS v4`：样式系统与设计令牌承载层。
- `TanStack Router`：路由组织、页面嵌套和导航结构。
- `TanStack Query`：服务端状态请求、缓存、轮询与失效控制。
- `Zustand`：本地 UI 状态、界面偏好和轻量会话状态。
- `Recharts`：图表和可视化组件的统一绘图库。
- `Clerk`：认证与账户相关能力。

## 职责边界

为避免后续技术栈混用，职责边界统一如下：

- 路由与页面组织：统一使用 `TanStack Router`。
- 服务端状态与缓存：统一使用 `TanStack Query`。
- 本地 UI 状态：统一使用 `Zustand`，不引入额外全局状态方案。
- 图表展示：统一使用 `Recharts`，保证行情与资产图表风格一致。
- 基础组件层：统一使用 `shadcn/ui` 与现有 `src/components/ui`。
- HTTP 请求与接口封装：沿用集中式 `lib/api` / `lib/market-api` 模式，不在页面内散落请求逻辑。

## 工程原则

### 类型优先

- 接口返回、表格数据、图表数据和状态枚举都应优先定义明确类型。
- 页面层不直接消费不稳定的原始响应结构，应通过适配层转换为页面模型。

### 组件复用优先

- 通用卡片、指标块、表格工具栏、状态徽标、空态与错误态组件优先复用已有组件。
- 新增组件前先检查是否可在现有组件上扩展，而不是复制模板页面。

### 配置驱动优先

- 导航、看板卡片、表格列、状态映射、风险等级等优先设计为配置驱动。
- 避免在多个页面硬编码相同业务文案、颜色语义或状态判断。

### 可观测性优先

- 核心业务页必须显式处理加载态、空态、错误态、无权限态和延迟态。
- 行情、资产、任务、日志相关视图必须能体现更新时间、刷新机制或数据时效信息。

### 收敛优先于扩散

- 现有模板中的通用演示模块只能在确认适合 Swan-Admin 语义后保留。
- 不再继续扩展与量化运营后台无关的示例页面、示例实体和占位业务。

## 数据与页面实现约束

### 接口适配层集中管理

- 页面组件不直接拼接复杂接口字段。
- 列表页、图表页和看板页应通过集中适配层将接口转换为稳定的前端数据结构。
- 业务字段命名在进入页面层前统一，不让不同接口命名差异扩散到组件层。

### 准实时优先

- 行情和资产展示默认优先支持轮询或准实时刷新。
- 是否采用推送机制可以在后续阶段评估，但页面结构必须预留更新时间和刷新状态表达。

### 数值组件必须完整表达状态

所有重要数值组件至少需要考虑：

- 加载中
- 无数据
- 数据延迟
- 请求失败
- 风险或异常标记

### 表格和日志能力保持一致性

- 任务、日志、资产明细等列表类页面应尽量复用统一表格交互模式。
- 搜索、筛选、排序、分页和批量操作行为需要保持一致认知。

## 与现有仓库结构的关系

当前仓库已经具备以下可复用基础：

- `dashboard`：可演进为运营工作台。
- `squirrel`：可演进为行情监控主模块。
- `tasks`：可演进为任务监控模块。
- `logs`：可演进为日志审计模块。
- `settings`：可承载后台配置和展示偏好。

当前仓库中存在的模板残留模块、命名和示例数据，应在后续实现中持续收敛，而不是默认保留。

## 与设计文档的关系

`specs/design.md` 负责视觉风格、色彩、字体、交互质感和页面气质。

本文件负责说明 Swan-Admin 应该用什么前端栈实现、每种技术负责什么，以及实现时必须遵守哪些工程边界。
